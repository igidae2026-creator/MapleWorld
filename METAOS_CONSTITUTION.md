# METAOS Constitution

MapleWorld operates under a repository-level autonomous governance skeleton.

## 1. System Identity

MapleWorld is a Codex-native autonomous architecture and operation evolution system for a MapleLand-class MMORPG under MapleStory Worlds constraints.

The system exists to improve actual player experience, not merely to accumulate assets, metrics, or documents.

## 2. Highest Invariants

- MapleStory Worlds-native runtime constraints only
- no external backend
- server-authoritative gameplay
- explicit authority, save, rollback, replay, room, world, channel, and instance boundaries
- deterministic offline evaluation and repeatable progress tracking
- canon locks and route variance must not be normalized away
- top-level skeleton first; no replace-soon lower governance scaffolds

## 3. Delivery Objective

The repository must evolve toward MapleLand-class player feel with strong first-session readability, hour-one retention pressure, day-one return intent, stable economy coherence, and durable social density.

## 4. Authority Order

Authority flows in this order:

1. `GOAL.md`
2. `METAOS_CONSTITUTION.md`
3. layer checklists and patch method
4. repository implementation files
5. generated reports and prompt assets

Lower layers may elaborate higher layers, but may not override them.

## 5. Layer Architecture Principle

MapleWorld uses a three-layer checklist system to operationalize this constitution.

- Layer 1 defines objective constraints.
- Layer 2 defines module responsibilities enforcing those constraints.
- Layer 3 maps those responsibilities to concrete repository files and machine-visible outputs.

The relationship is:

`METAOS_CONSTITUTION.md` -> `CHECKLIST_LAYER1_목표조건.md` -> `CHECKLIST_LAYER2_모듈책임.md` -> `CHECKLIST_LAYER3_REPO매핑.md` -> `CHECKLIST_METHOD_패치.md`

If any layer fails to preserve constitutional invariants, player-experience gates, or authority boundaries, the architecture is incomplete.

## 6. Mutation Law

When governance must change:

- update the top-level authority first
- update coverage and conflict records
- then update lower-level implementation or references

All lower-layer governance mutation must follow the canonical patch protocol in `CHECKLIST_METHOD_패치.md`.

## 7. Artifact Competence Principle

MapleWorld outputs must satisfy a competence floor rather than merely existing as structurally valid artifacts.

That floor must include:

- generic competence
- domain objective performance
- usable player-facing quality under MapleStory Worlds constraints

Validation, repair, simulation, and score selection must enforce this floor. Quality must not depend on a lucky one-off generation.

## 8. Quality Gradient Principle

MapleWorld must maintain a measurable improvement gradient across repeated iterations.

Single isolated wins are insufficient. The system must prefer changes that improve or preserve:

- `first_10_minutes`
- `first_hour_retention`
- `day1_return_intent`
- economy coherence
- route variance
- social density

If repeated cycles fail to improve quality or repeatedly trade player feel for structural gains, the repository must trigger bottleneck re-selection, architectural mutation, or rule repair.

## 9. Optimization Law

Optimization targets are valid only if they preserve player-experience gates, canon locks, authority safety, and deterministic repairability.

## 10. Outer-Autonomy Law

MapleWorld autonomy must expand beyond already-included repository scope.

It is insufficient for the system to only iterate on files and materials that are already inside the governed loop. The system must evolve toward an outer automation layer that can:

- inspect newly arriving materials and references
- judge scope fit, authority fit, and upgrade value automatically
- reject out-of-scope or low-value materials without human sorting
- elevate qualified materials into the governed repository flow with minimal or no human intervention

The autonomy target is reached only when normal human intervention adds little or no additional quality beyond what the system already produces on its own.

## 11. Anti-Drift Law

Conversation is not governance authority.

Persistent authority must live in repository files and audits, not in transient chat memory.
